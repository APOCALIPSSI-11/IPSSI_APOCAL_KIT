# T-26.3 — Frontend : route `/signup-enseignant` + formulaire dédié

**US** : US-26 · **Assigné** : Hugo · **Estim.** : 2h · **Stack** : React + TypeScript + React Router + Tailwind

---

## 1. Objectif

Ajouter un parcours d'inscription **dédié aux enseignants**, distinct de l'inscription étudiant existante (`/signup`) :

- une nouvelle route publique `/signup-enseignant` ;
- une page avec un formulaire propre au rôle enseignant (établissement + code d'invitation en plus des champs classiques) ;
- l'appel à l'endpoint backend `POST /api/accounts/signup-enseignant/` (T-26.2).

## 2. Dépendance

Cette tâche **dépend de T-26.2** (endpoint backend). Au moment de coder, le contrat exact de l'endpoint n'est pas figé dans le dépôt. J'ai donc retenu un contrat plausible à **aligner avec le backend** :

```
POST /api/accounts/signup-enseignant/
{
  "email": string,
  "password": string,
  "first_name"?: string,
  "last_name"?: string,
  "school": string,
  "invite_code": string
}
→ 201 { User }
```

> Si T-26.2 utilise d'autres champs (ex. `discipline`, `matricule`), il suffit d'ajuster l'état du formulaire et le type de `signupEnseignant`.

## 3. Choix d'implémentation

### Pourquoi une page séparée plutôt qu'un champ « rôle » sur `SignupPage` ?
Le parcours enseignant a ses propres contraintes (code d'invitation **obligatoire**, établissement) et un **endpoint distinct**. Ajouter un formulaire « à tiroirs » sur la page existante la rendrait plus dure à tester et à maintenir. Deux pages côte à côte = deux parcours lisibles, avec un lien croisé entre elles.

### Réutilisation des conventions du projet
Pour rester cohérent avec le code existant, je réutilise :
- le **template visuel** de `SignupPage.tsx` (classes `card`, `input`, `btn-primary`, gestion `allow_signups`) ;
- `getApiErrorMessage(err, fallback)` pour afficher les erreurs de validation DRF par champ ;
- le pattern **auto-login après inscription** (via `login()`), déjà utilisé par `signup()` ;
- l'alias d'import `@/`.

### Redirection après succès
L'étudiant est redirigé vers `/upload`. Pour l'enseignant, je redirige vers `/dashboard` (espace plus pertinent pour un profil enseignant). À confirmer selon le parcours produit voulu.

## 5. Étapes de développement

1. **API** : ajouter `signupEnseignant()` dans `api/auth.ts` (calquée sur `signup()`, endpoint et champs différents).
2. **Page** : créer `SignupEnseignantPage.tsx` en repartant de `SignupPage.tsx`, puis ajouter les champs `school` et `invite_code` et changer le titre/textes.
3. **Route** : dans `App.tsx`, importer la page et déclarer `<Route path="signup-enseignant" ... />` juste après `signup`.
4. **Lien croisé** : ajouter un lien « Inscription étudiant / enseignant » entre les deux pages (fait dans le JSX).
5. **Tests** : rendu des champs, cas succès (appel API + redirection), cas erreur (message affiché, pas de redirection).

## 6. Critères d'acceptation

- [ ] `/signup-enseignant` est accessible sans être connecté.
- [ ] Le formulaire exige email, mot de passe (≥ 8), établissement, code d'invitation et l'acceptation CGU.
- [ ] La soumission appelle `POST /api/accounts/signup-enseignant/` avec les bons champs.
- [ ] En cas de succès : auto-login + redirection.
- [ ] En cas d'erreur backend : message lisible affiché, pas de redirection.
- [ ] Inscriptions fermées (`allow_signups=false`) : écran « Inscriptions fermées ».
